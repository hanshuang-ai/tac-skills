# 与概要设计的一致性检查

派生 LLD 前后必须检查与上游 `01-概要设计.md` 与 `00-业务与领域设计.md` 的一致性。

## 1. 进入派生前（HLD 完整性反查）

- [ ] `01-概要设计.md` 已落盘到 `persistent-assets/design/_baseline/`，用户已审批通过
- [ ] 技术栈选型 ≥ 2 备选 + 推荐 + 反驳替代
- [ ] 架构风格 + 反驳替代风格段落存在
- [ ] C4 Context + Container 双图存在
- [ ] 模块清单 ≥ 1 行，每行五字段填齐（模块/职责/上下文/上游/下游）
- [ ] 模块间通信表全部标注通信方式 + 失败处理
- [ ] 外部依赖表全部填齐五要素（系统/协议/版本/SLA/降级）
- [ ] 跨切面 6 项全部回应（具体策略或"不适用：原因"）
- [ ] NFR 达成路径表覆盖 PRD 全部非功能需求

任一项 ✗ → 拒绝执行，回上游 `tac-hld` 补全。

## 2. 派生后（HLD → LLD 派生一致性反查）

落盘后 + 自审前执行：

- [ ] **模块清单 = 模块详述**：HLD 模块清单 ⊆ LLD §4.x 详述（不漏）；LLD §4.x ⊆ HLD 模块清单（不多）
- [ ] **依赖关系一致**：HLD §6 模块间通信中每条边在 LLD 各模块 §4.x.2 依赖清单中双向出现
- [ ] **外部依赖一致**：HLD §7 外部依赖表全部在某模块 §4.x.2 中被消费
- [ ] **跨切面落地**：HLD §8 跨切面策略中的具体技术（日志框架、监控指标、鉴权方式）在 LLD 命名/埋点/接口契约中体现
- [ ] **存储一致**：HLD §9.1 聚合根→存储映射与 LLD §4.x.7 数据模型表名/字段一致
- [ ] **NFR 落地**：HLD §11 每条 NFR 在某模块 §4.x.9 可观测性中可被度量

## 3. 业务领域 → LLD 派生一致性（递归补检）

- [ ] **业务规则 → 行为规约**：业务领域 §8 每条 R# 在 LLD 某模块 §4.x.4 内部行为规约或 §4.x.5 状态机中体现
- [ ] **聚合根 → 数据模型**：业务领域 §6 每个聚合根字段在 LLD §4.x.7 中能找到
- [ ] **领域事件 → 时序图**：业务领域 §7 每个领域事件在 LLD 某模块 §4.x.6 时序图中触发

## 4. 内部完整性自查

- [ ] 模块详述章节数 = HLD 模块数（不漏不多）
- [ ] 每个模块都有：概述/依赖/接口/规约/状态机/时序图/数据模型/并发/可观测性/风险 全 10 个子章节
- [ ] 每个对外接口都有六要素（接口类型/签名/前置/后置/错误码/幂等）
- [ ] 状态机标注 entry/exit/非法事件处理
- [ ] 时序图覆盖全部错误码触发路径
- [ ] 全局错误码表与各接口错误码列表互相一致

## 5. 维护模式增量反查

仅对影响清单中涉及的元素核查，不重跑全集。

## 6. 检查执行方式

- 自动化最小化：用 grep/diff 校验文件存在与章节锚点
- 语义检查：通读派生与追溯表，逐行核对
- 模板内置追溯表（LLD 模板末尾的"派生与追溯"章节）填齐它就完成大半检查
