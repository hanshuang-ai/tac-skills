# 质量证据审核参考

准出审核是证据审计。以下维度来自原质量交付 checklist，现作为 `tac-release-gate-check` 的证据判断参考。

| 维度 | PASS 证据示例 | 常见 BLOCKED |
|---|---|---|
| 功能完成度 | 验收记录、单测/集成测试、UI 自动化、代码路径、接口验证 | 有 in-scope 项但没有实现/测试/验收证据 |
| 性能要求 | benchmark、Profiler、Perfetto、压测报告、监控截图 | 有性能标准但没有负载条件、设备条件或结果 |
| 代码质量 | 构建、单测、lint、静态扫描、覆盖率、Review approval | 只声明“已自测”，没有命令或报告 |
| 安全准出 | 安全扫描、依赖漏洞报告、权限审计、代码审计、合规审批 | 涉及敏感数据/权限但没有扫描或审批 |
| 架构合理性 | 依赖图、模块边界审查、架构 lint、设计偏离记录 | 核心架构变更但没有设计同步或评审记录 |
| 可观测性 | 日志字段、指标查询、Trace、告警规则、发布观察项 | 关键链路无日志/指标/告警证据 |
| 兼容性 | 设备/系统矩阵、升级测试、协议兼容测试、迁移验证 | 涉及兼容/迁移但没有矩阵或回滚验证 |

## 审核要点

### 功能完成度

- PRD in-scope 是否都有实现和验证证据。
- 业务规则是否有代码落点和测试或验收记录。
- 接口契约、错误码、状态机、异常路径是否可验证。
- 历史回归是否被测试用例或验收项锁定。

### 性能要求

- 性能结果必须包含阈值、设备/环境、负载条件和数据。
- 只有“性能正常”或“体验流畅”不能 PASS。
- 缺少上游阈值时，不能编造阈值；按标准要求标记 BLOCKED 或 OPEN 风险。

### 代码质量

- 构建、测试、lint、静态扫描应引用实际命令和结果。
- 新增公开 API、状态转换、核心业务逻辑应有测试证据。
- Review approval 可作为过程证据，但不能替代构建/测试结果。

### 安全准出

- 涉及鉴权、权限、敏感数据、第三方依赖时，缺少扫描或审批通常是 BLOCKED。
- 人工安全确认必须记录确认人、范围和风险接受内容。

### 架构合理性

- 模块边界、依赖方向、跨上下文调用要能从代码或依赖图验证。
- 偏离 HLD/LLD 时，必须有决策记录或明确风险接受。

### 可观测性

- 关键日志必须能定位问题，不泄露敏感信息。
- 指标、Trace、告警要有查询入口或配置路径。
- 发布级准出应包含发布后观察指标。

### 兼容性

- 兼容性结论必须绑定矩阵：系统版本、设备、协议版本、历史数据或渠道。
- 涉及数据迁移时，升级验证和回滚/恢复策略是硬证据。

## 结论规则

- 硬性标准缺证据：`BLOCKED`。
- 硬性标准证据不满足：`FAIL`。
- 建议项缺证据：记录风险；若风险已被明确接受，可支持 `有条件准出`。
- 标准确实不适用：`N/A`，必须说明原因。
