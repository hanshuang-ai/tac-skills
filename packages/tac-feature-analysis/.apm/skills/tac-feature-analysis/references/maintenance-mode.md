# 维护模式（Maintenance Mode）

> 当 PRD 已经跑过本 Skill 一次（4 份项目级 baseline 产物已落盘且 commit），后续 PRD 变化（澄清 / 修改 / 新增 / 废弃）应该走**维护模式**，而非重跑全集。
>
> 触发判断：用户在 Step 0 提供了「迭代来源」字段（上一版 feature-slug + 该版本最末次的 commit hash）。
>
> **路径模型**：4 份产物位于 `persistent-assets/design/_baseline/` 项目级固定路径，维护模式默认在这些文件上**就地增量修订**；旧版本保留由 git history / branch / tag 承担，不再通过新建目录复制实现。

## 维护模式的 5 类变化

| # | 变化类型 | 表现 | 处理方向 |
|---|---------|------|---------|
| 1 | **澄清** (Clarify) | 原有 ASSUMPTION / 模糊点被 PRD 作者明确（例：M1 "装机配额仅管控已安装数" 被官方确认） | 移除 ASSUMPTION 标注，把模糊语义替换为明确语义；不改变结构 |
| 2 | **修改** (Modify) | 原有需求语义改变（例：装机配额从 12 改为 20；冷启动 KPI 从 3s 改为 2s） | 改动具体值或规则文本；状态机 / 接口 / 用例的判据量级随之更新 |
| 3 | **新增** (Add) | PRD 增加章节 / 能力 / 错误码 / 状态（例：新增"应用收藏"） | 在既有 BC 内追加聚合根/规则/接口/用例；或新建 BC（罕见） |
| 4 | **废弃** (Remove) | PRD 删除某章节或能力 | 把对应元素从 in-scope 移到 out-of-scope；保留历史描述但加标注；删除关联接口 / 错误码 / 用例（保留 changelog） |
| 5 | **重命名** (Rename) | 业务术语调整（例：CP App → Third-Party App） | 全文替换 + 在 Stage 1 §9 Ubiquitous Language 加"原术语 → 新术语"映射 |

## 维护模式 5 步流程

### M0. 收集变化清单（替代首次模式 Step 0 的 PRD 通读）

必填字段（**追问模板**）：

```
为开展维护模式分析，请提供：

【迭代来源】：上一版 feature-slug + 最末次 commit hash（例：appstore-srs-v1.2 @ 45de47d）
【新 PRD 路径】：<新版本 PRD 的相对路径，必须可读>
【变化清单】：每条按以下格式列出，按类型分组
  - [澄清] 原 ASSUMPTION M1 "装机配额管控范围" → 仅管控已安装数；下载未安装不计入配额
  - [修改] 装机配额值 → 12 改为 20
  - [新增] 应用收藏功能（PRD §2.17，能力归 BC-AppCatalog 内）
  - [废弃] §2.6 FNV Diagnostic（合规调整）
  - [重命名] 全文 "CP App" → "Third-Party App"
```

收齐后确认 `persistent-assets/design/_baseline/` 下 4 份 baseline 文件齐全，进入 M0.1 维护策略选择。

### M0.1 维护策略选择（与用户确认）

| 策略 | 适用 | 操作 |
|------|------|------|
| **A. 就地增量修订**（推荐，默认） | 变化体量小到中等（澄清 / 修改 / 在既有 BC 内新增） | 直接在 `persistent-assets/design/_baseline/{00-业务与领域设计,01-概要设计,02-详细设计,03-准出标准设计}.md` 4 份文件上追加变更记录 + 修改对应章节；旧版本对照走 `git diff <iter-from-hash>` |
| **B. 重新派生覆盖** | PRD 重大方向调整（v1.x → v2.0），BC 大变；既有 baseline 已不能作为可靠上游 | 走首次模式 Step 0~5，覆盖式重写 4 份 baseline；旧版本对照走 `git log persistent-assets/design/_baseline/` 与 branch/tag |

> 默认推荐 A；只有当"新增"覆盖**全新 BC**（不在既有任何 BC 内）+ 既有 baseline 多数章节失效时才升级到 B。
>
> **注**：由于 baseline 路径固定，不再存在「新建目录复制并修订」的中间策略；如需保留旧版本快照对照，请使用 `git tag baseline-v<旧版本>` 在维护开始前打标签。

### M1. 影响矩阵（Impact Matrix）

按变化清单逐条分析对 4 份产物的影响，输出影响矩阵给用户审阅：

```markdown
| 变化 # | 类型 | 来源 | Stage 1 影响 | Stage 2 影响 | Stage 3 影响 | Stage 4 影响 |
|-------|------|------|-------------|-------------|-------------|-------------|
| 1 | 澄清 M1 | PRD §2.2 | §3.1 第 4 项移除 ASSUMPTION + §8 R8 文本精确 | 无 | §4.2.4 行为规约更新 | AC-DL-FN-05 / AC-RU-08 判据更新 |
| 2 | 修改装机配额 12→20 | PRD §2.3.2 | §8 R8 配额值更新 | §11 NFR 装机配额行更新 | §4.2.4 / §4.2.7 数据模型默认值 | AC-RU-08 / AC-PF-14 判据更新 |
| 3 | 新增应用收藏 | PRD §2.17 | §5.4 BC-AppCatalog 能力清单加 2 行 + §6.1 Application 加 favorited 字段 + §8 加 R35 / R36 + §10 加新流程 | §5 feature-mine 职责扩展 + §9.1 加 favorites 表 | §4.5 feature-mine 加接口 + §4.14 加 favorites 数据模型 | §3.5 feature-mine 加用例 + §5 R35/R36 用例 + §9.x 派生 |
| 4 | 废弃 FNV | PRD §2.6 | §3.2 加 out-of-scope；§7 删除 `售后日志已上报` 事件；§8 R31 删除 | §6 删除 FNV 通信；§8.2 监控可观测段更新 | §4.6 feature-telemetry 删除 fnvLog 接口 + 错误码 E_TEL_INTERNAL 范围 | AC-TEL-FN-02 / AC-RU-31 删除 |
```

**影响矩阵自审**：

- [ ] 每条变化的"类型"明确（澄清/修改/新增/废弃/重命名）
- [ ] 每条变化指向具体 PRD 章节
- [ ] 4 个 Stage 列至少一个有影响（否则该变化要么是无效变化，要么影响识别不全）
- [ ] 跨阶段一致性预演：Stage 1 改了 BC，Stage 2-4 必须有对应改动；否则派生链断

用户审阅影响矩阵 → 确认无遗漏 → 进入 M2。

### M2. 局部修改（按 Stage 顺序）

按 M1 影响矩阵逐文件改。**严格只改矩阵中标注的章节**，不顺手改其他内容。

每份文件**头部"派生自"块下方追加变更记录段**（首次写则新建该段，已有则追加行）：

```markdown
> 派生自：...
> 文档版本：v1.0 · 2026-05-06
>
> ## 变更记录
>
> | 日期 | 变化来源 | 类型 | 影响章节 | 简述 |
> |------|---------|------|---------|------|
> | 2026-05-10 | PRD §2.2 (M1 澄清) | 澄清 | §3.1 / §8 R8 | 装机配额仅管控已安装数；下载未安装不计入 |
> | 2026-05-10 | PRD §2.3.2 | 修改 | §8 R8 / §11 NFR | 配额值 12 → 20 |
> | 2026-05-12 | PRD §2.17（新增） | 新增 | §5.4 / §6.1 / §8 R35-R36 / §10 | 应用收藏功能 |
> | 2026-05-12 | PRD §2.6（废弃） | 废弃 | §3.2 / §7 / §8 R31 | FNV Diagnostic 移出 in-scope |
```

**每条变更记录的强约束**：
- "影响章节"列必须与影响矩阵 M1 中本文件对应列严格一致
- 类型为"修改"时，应在所改章节内联用脚注或行内注引用 changelog 行（例：`R8: 装机配额 ≤ 20（v1.2.1 修改自 12）[changelog 2]`）；如果脚注会破坏可读性，至少在派生表中可追溯

### M3. 跨阶段一致性反查（增量版）

**只检查影响矩阵中涉及的元素**，不重跑全集。

按 `cross-stage-consistency.md`，对受影响的 §3.3 / §4.3 / §4.4 派生一致性逐条核实：

- 新增 BC → 模块映射、聚合根 → 存储映射 等是否齐全
- 废弃元素是否在 4 个 Stage 中全部移除 / 移到 out-of-scope
- 修改值是否 Stage 1 → 4 同步（避免出现 Stage 1 写 20、Stage 2 仍写 12）

### M4. 双闸门（自审 + 用户审）

**每份文件**做 5 类自审，但只对变化章节判：

- Placeholder：变化处不留 TODO/TBD
- Consistency：与未变化章节不矛盾
- Scope：变化未越界（业务领域阶段不写技术；HLD 阶段不写代码）
- Ambiguity：变化措辞清晰
- Stage-specific 第 5 类：按既有规则

通过后给用户审 prompt（**重要**：用 prompt 突出"维护模式 + 变化清单 + 影响章节"）：

```
> Stage <N> 已按维护模式更新 `persistent-assets/design/_baseline/<file>.md`（未 commit）。
>
> 变化清单（共 N 条）：[列表]
> 影响章节：[列表]
>
> 已通过 5 类自审（只对变化章节判）。
> 跨阶段一致性反查（增量）通过。
>
> 请审阅，确认后由你 commit（建议在 commit message 中显式列出所有变化条目）。
```

### M5. 交棒

四份产物全部维护通过后，按变化体量推荐下游路径：

- **小变化（澄清 / 单条修改）**：直接通知相关 Feature 负责人 / 不需要新 spec
- **中变化（多条修改 / 新增能力点）**：建议对受影响 Feature 跑单 Feature 入口工具单点更新（如本仓库 `tac-feature` / Speckit 体系等价工具）；引用更新后的 4 份产物作为约束
- **大变化（新 BC / 重大重命名）**：可能整段下游重新跑规格化工具（如 `speckit-specify` 或所在项目等价工具）

## 维护模式与首次模式的关键差异

| 维度 | 首次模式 | 维护模式 |
|------|---------|---------|
| Step 0 输入 | PRD 路径 + feature-slug + 范围 | 上述 + **迭代来源 + 变化清单** |
| 派生主线 | PRD → 全集 | 旧产物 + 变化清单 → 局部修改 |
| 章节生成 | 全部新写 | 仅改影响章节 |
| 自审 | 对全文 5 类自审 | 仅对变化章节 5 类自审 |
| 用户审 | 整篇审 | 按变化清单审 + 复看影响章节 |
| Commit | 4 份各自 commit（见硬规则 5） | 推荐**单 commit** 含 4 份所有改动，message 列变化清单 |
| 派生与追溯表 | 全表新写 | 末尾追加 changelog 引用行；未变章节追溯不动 |

## 自审专项：变化记录完整性

每次维护模式跑完后追加自审第 6 类：

- [ ] 4 份产物头部均含「变更记录」段，新增条目与本次变化清单 1:1
- [ ] 每条变更记录的"影响章节"列在该文件中确实有对应改动（非空写）
- [ ] 影响矩阵中的每个 Stage 列影响 = 实际改动章节（不漏不多）
- [ ] 4 份产物中所有"修改"类变化的具体值（如配额、KPI 阈值）4 份保持一致

任一项 ✗ → 回 M2 补全。

## 反模式（不允许的做法）

| 反模式 | 后果 | 纠偏 |
|-------|------|------|
| 在维护模式下重写未变章节 | 历史 commit 难以 diff、审阅成本爆炸 | 严格只改影响矩阵列出的章节 |
| 不更新派生与追溯表 | 后续无法追溯"v1.3 的某个用例为什么存在" | 每次新增 / 修改 / 删除元素都要在 §11 / §13 / §5 / §9 派生表中加 changelog 行 |
| 跳过影响矩阵直接改文件 | 改漏 Stage 2/3/4 中的连锁影响 | 强制先输出影响矩阵给用户审 |
| 把"修改" 当 "新增" 处理 | 旧值残留与新值并存 | 修改类变化必须**替换**原值，不是追加 |
| 单文件 commit 拆 4 个 | 维护一组逻辑变化分 4 commit 后难以一致回滚 | 推荐单 commit 含 4 份，message 在 Description 列出所有变化条目 |

## 与首次模式的代码复用

维护模式与首次模式共享：

- `cross-stage-consistency.md`（跨阶段一致性检查器，增量适用）
- `dld-reviewer-prompt.md` / `acceptance-reviewer-prompt.md`（子代理深审，对变化部分仍可派）
- 各 stage reference 的派生方法、必填字段、自审 5 类

不同处：

- 维护模式额外读本文件
- 维护模式跳过 PRD 全集通读，只读 PRD 变化章节
- 4 份 templates 头部的"变更记录"段在维护模式下必填
