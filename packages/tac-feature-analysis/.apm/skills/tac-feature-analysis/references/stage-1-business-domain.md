## 3. 派生方法（PRD → 业务领域）

| 章节 | 从 PRD 哪里推 | 方法 |
|------|--------------|------|
| 业务背景与目标 | PRD 开篇/概述 | 摘录关键业务诉求与商业目标，剔除实现描述 |
| 利益相关者与角色 | PRD 用户故事/使用场景 | 抽取所有 actor，分内部用户 / 外部用户 / 系统角色 |
| 范围声明 | Step 0 的范围声明 | 直接落入，不二次裁剪 |
| 子域分类 | PRD 功能模块 | 按"业务差异化竞争 / 必要支撑 / 行业通用"判核心/支撑/通用 |
| 限界上下文 | PRD 功能模块 + 子域 | 每个限界上下文必须能用一句话回答"它的语言模型是什么" |
| 上下文映射 | PRD 模块间交互描述 | 标注关系类型：共享内核 / 客户-供应方 / 防腐层 / OHS / Conformist |
| 聚合根与实体 | PRD 名词 + 状态描述 | 找到"有生命周期 + 状态变化 + 一致性边界"的对象 = 聚合根 |
| 领域事件 | PRD 动词短语 + "通知/触发/导致"句式 | 过去式命名（如 `订单已支付`），避免命令式（如 `支付订单`） |
| 业务规则与不变式 | PRD "必须 / 不能 / 至少 / 至多 / 仅当" 句式 | 提炼为可被代码 assert 的命题 |
| Ubiquitous Language | 全篇关键名词 | 业务术语 + 一句话定义 + 是否是聚合/实体/值对象 |
| 业务流程 | PRD 用户故事 | 至少画出每个核心域 happy path + 1 条关键异常路径 |
| 能力清单（§5.4） | PRD 功能描述 + 限界上下文 | 按 BC 组织能力单元，每条能力填齐六要素：输入 / 输出 / 规则（关联 R#）/ 状态（关联聚合根状态）/ 异常（拒绝条件或失败码）/ 依赖（下游 BC 或系统能力）。能力按用户视角的可感知功能粒度命名 |
| 场景-行为联动表（§10.3） | PRD"必须 / 仅当"句式 + 业务规则 | 识别每个关键场景变化对 UI / 接口 / 缓存 / 权限四类行为的连锁影响。至少覆盖：授权状态变化 / 网络可用性变化 / 座舱场景切换（高并发 / 走行限制 / STR）/ 应用上下架 / 协议版本变更 |

## 6. 自审检查

执行通用 4 类（Placeholder / Consistency / Scope / Ambiguity）+ Stage 1 专属第 5 类：

**第 5 类 — Domain Purity（领域纯净度）：**

- 是否提到了具体技术栈（如 "Room"、"Retrofit"、"PostgreSQL"）？→ 删除
- 是否提到了具体存储/接口路径（如 "table users"、"GET /api/orders"）？→ 删除
- 是否描述了 UI 控件/页面布局（如 "卡片列表展示"）？→ 删除
- 是否使用了"模块"、"服务"、"组件"等技术划分词替代业务概念？→ 改回业务术语
- 限界上下文的语言模型描述是否真的从业务视角写？（自检：把这句话给业务方看能否理解）

**自审失败行动**：inline 修复，不需要二审。

## 8. 常见踩坑

| 踩坑 | 表现 | 纠偏 |
|------|------|------|
| 限界上下文 = PRD 模块 | 直接拷 PRD 模块名当限界上下文 | 限界上下文按"语言一致性"划分，可能合并多个 PRD 模块或拆分单个模块 |
| 聚合根膨胀 | 一个聚合根包含 5+ 实体和大段子树 | 拆为多个聚合根 + 引用，单个聚合根不变式不应跨多组实体 |
| 领域事件命名不规范 | "支付订单"、"创建订单" | 统一过去式："订单已支付"、"订单已创建" |
| 把"功能"当"领域" | 章节里全是"如何做" | 业务领域只描述"是什么 / 何时发生 / 受什么约束"，不描述"怎么实现" |
| 跨上下文交互含糊 | "A 和 B 交互" | 必须标注 DDD 模式：客户-供应方 / 共享内核 / 防腐层 / OHS / Conformist / 分离方式 |
