# 场景指南（Scenario Guide）

> 来源: SKILL.md §2
> 版本: v4.1
> 用途: 0-1 / 1-N / Bugfix 三种场景详细拆分策略（含迭代标识批次区分）

---

## 快速对照

| 维度 | 0-1 | 1-N | Bugfix |
|------|-----|-----|--------|
| **定义** | 从零构建 | 在既有产品上迭代 | 修复缺陷 |
| **输入** | 全量设计 | 增量设计 | Bug Report |
| **范围** | 全量功能树 | 增量 Story | 修复点+防御 |
| **Story 数** | 5-50+ | 1-10 | 1-3 |
| **迭代标识** | 当前批次（全同） | 既有保留，新增用新标识 | 沿用原标识 |
| **约束重点** | 技术栈/平台基线 | 兼容性/不破坏既有 | 修复范围最小化 |
| **验收偏好** | inline_gwt | reuse_baseline+GWT | 回归用例 |

## 0-1 场景

### 判定条件
用户说"新项目""从零开发""全新模块"，workspace 中无对应代码。

### 拆分步骤
1. 从 PRD 信息架构识别模块
2. 每个模块下识别功能组（每个功能组 = 一个 Story）
3. 全部 Story 迭代标识设为 `IT-V1.0`（绑定 SRS V1.0 版本号）
4. 全部 Story 日期设为当前日期
5. 填充九字段，标记并行

### 迭代标识示例
0-1 全量拆分迭代标识 = `IT-V1.0`。所有 20 个 Story 的迭代标识一致。同份需求重拆时迭代标识仍为 `IT-V1.0`，只有日期变化。二期若新增功能，新 Story 迭代标识 = `IT-V1.1`。

## 1-N 场景

### 判定条件
用户说"新增功能""迭代""二期"，workspace 中有既有代码。

### 拆分步骤
1. 先读取 `decomposition_index.yaml` 获取既有 Story 列表及其迭代标识
2. 确认变更影响面
3. 在功能树上标注 `[IT-V1.0]`（既有）和 `[IT-V1.1 新增]`（增量）
4. 只拆分增量部分
5. 增量 Story 迭代标识 = 新批次标识（如 `IT-V1.1`）

### 迭代标识示例
```
一期（IT-V1.0）:
  RQ-HOM-BNR-01  Banner展示    迭代标识: IT-V1.0  ← 已实施
  RQ-HOM-REC-01  应用推荐       迭代标识: IT-V1.0  ← 已实施

二期（IT-V1.1）:
  RQ-HOM-SRC-01  搜索功能       迭代标识: IT-V1.1  ← 新增
  RQ-HOM-SRC-02  搜索建议       迭代标识: IT-V1.1  ← 新增
```

## Bugfix 场景

### 判定条件
用户说"修复""bug""crash""报错"，输入为缺陷单。

### 拆分步骤
1. 分析缺陷单：现象/预期/复现
2. 定位受影响 Story
3. 拆出修复 Story
4. 迭代标识沿用原始批次，日期 = 当前修复日期
