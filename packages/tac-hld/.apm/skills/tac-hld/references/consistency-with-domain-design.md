# 与业务与领域设计的一致性检查

派生 HLD 前后必须检查与上游 `00-业务与领域设计.md` 的一致性。任一不通过必须先回上游补全或本阶段修订。

## 1. 进入派生前（业务领域完整性反查）

- [ ] `00-业务与领域设计.md` 已落盘到 `persistent-assets/design/_baseline/`，用户已审批通过
- [ ] 限界上下文表 ≥ 1 行，每行包含一句话语言模型
- [ ] 上下文映射图存在且每条边标注关系类型
- [ ] 聚合根 ≥ 1 个，每个聚合根含不变式
- [ ] 领域事件 ≥ 3 个，全部过去式命名
- [ ] 业务规则 ≥ 5 条，全部为可断言命题
- [ ] Ubiquitous Language 覆盖全部聚合根/实体
- [ ] 派生与追溯表存在，PRD 元素一一对应

任一项 ✗ → 拒绝执行，回上游 `tac-domain-design` 补全。

## 2. 派生准备

- [ ] 已读项目入口文件（`PROJECT.md` / `AGENTS.md` / `CLAUDE.md` / `GEMINI.md` / `README.md` 之一）中本 PRD 命中场景的专项规范清单；未命中则按通用最佳实践处理
- [ ] 已识别项目既有技术栈与架构约束（按检测顺序：入口文件 Active Technologies 段 / `gradle/libs.versions.toml` / `package.json` 等；避免重复造轮子）

## 3. 派生后（业务领域 → HLD 派生一致性反查）

落盘后 + 自审前执行：

- [ ] **限界上下文 → 模块**：业务领域每个 BC 在 HLD 模块清单中有 ≥ 1 个对应模块
- [ ] **聚合根 → 存储**：业务领域每个聚合根在 HLD §9.1 表中有存储映射
- [ ] **领域事件 → 通信**：业务领域每个领域事件在 HLD §6 模块间通信表中能找到对应行（同步/异步/共享存储）
- [ ] **业务规则 → 一致性策略**：业务领域涉及跨聚合的业务规则在 HLD §9.2 中有显式一致性级别声明
- [ ] **术语一致**：HLD 模块/存储命名沿用业务领域 Ubiquitous Language（不引入新业务术语）

任一项 ✗ → 回业务领域补全或回 HLD 修订（看缺口在哪一侧）。

## 4. 维护模式增量反查

仅对影响清单中涉及的元素核查，不重跑全集：

- 新增 BC → 模块映射、聚合根 → 存储映射 等是否齐全
- 废弃元素是否在 HLD 中全部移除 / 移到 out-of-scope
- 修改值是否业务领域 → HLD 同步（避免业务领域写 20、HLD 仍写 12）

## 5. 检查执行方式

- 自动化最小化：用 grep/diff 校验文件存在与章节锚点
- 语义检查：通读派生与追溯表，逐行核对
- 模板内置追溯表（HLD 模板末尾的"派生与追溯"章节）就是为这套检查准备的——填齐它就完成大半检查
