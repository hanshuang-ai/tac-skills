# 准出标准派生方法

## 核心原则

准出标准不是通用 checklist，而是从当前项目的业务、设计、代码、测试和治理约束中派生出的质量门禁。

每条标准都要回答：

| 问题 | 要求 |
|---|---|
| 它约束什么风险？ | 业务正确性、数据一致性、安全、性能、兼容、可靠性、可观测、发布回滚等 |
| 它属于哪个层级？ | 项目/系统级、版本发布级、需求/功能级 |
| 它是否硬性阻断？ | 硬性准出标准或建议检查项 |
| 如何证明通过？ | 命令、测试、代码路径、评审记录、监控信号、人工审批 |
| 来源是什么？ | 需求、业务领域设计、HLD、LLD、代码、CI、宪章或推理 |

## 项目/系统级

从长期约束中派生：

- 架构边界：模块职责、依赖方向、跨层调用限制。
- 核心业务不变量：状态流转、权限、账户/订单/资产等不可破坏规则。
- 安全合规：敏感数据、权限、认证、日志脱敏、第三方依赖。
- 性能基线：启动、页面响应、接口耗时、资源占用、并发能力。
- 兼容范围：系统版本、设备型号、协议版本、数据迁移。
- 可观测性：关键链路日志、指标、告警、问题定位信息。

## 版本发布级

从本次发布风险中派生：

- 发布范围完整：需求、缺陷、配置、依赖、数据变更已纳入发布说明。
- 缺陷门槛：P0/P1 为 0；P2 有 owner、规避方案和发布审批。
- 回归范围：受影响模块、主流程、历史高风险链路完成验证。
- 构建产物：版本号、签名、渠道、配置、制品可追溯。
- 灰度与回滚：灰度策略、回滚路径、兼容旧数据或旧协议。
- 线上监控：发布后观察指标、告警阈值、值班或响应机制。

## 需求/功能级

从需求和实现细节中派生：

- 验收条件：PRD/issue/用户故事中的 in-scope 项全部可验证。
- 主流程：核心路径行为、页面跳转、接口交互、数据写入。
- 异常流程：错误码、空态、网络失败、权限拒绝、重试和恢复。
- 边界条件：最大/最小值、重复提交、并发、幂等、历史数据。
- 接口契约：请求/响应字段、错误码、兼容性、调用方影响。
- 测试证据：单测、集成测试、UI 自动化、手动验收或 CI 报告。
