# 软件详细设计 — 派生方法

## 1. 输入清单

| 来源 | 文件 | 用法 |
|------|------|------|
| 必读 | `persistent-assets/design/_baseline/01-概要设计.md` | 模块清单（必须逐模块详述） |
| 必读 | `persistent-assets/design/_baseline/00-业务与领域设计.md` | 业务规则、聚合根、领域事件、术语 |
| 必读 | 项目编码规范与工程规范（`.specify/memory/*.md` / `persistent-assets/governance/**/*.md` / `CONTRIBUTING.md` / `ENGINEERING.md` / `ARCHITECTURE.md` 等；项目入口文件可显式声明位置） | 接口风格、错误码体系、命名约定；未命中则按通用 Android/Kotlin 风格（驼峰命名、kebab-case 资源、错误码 `E_<MODULE>_<KIND>_<SEQ>` 形式） |
| 按需 | 既有同类模块代码 | 参考接口风格与命名 |
| 维护模式必读 | 上一版 `02-详细设计.md` + 变化清单 | 增量修改（仅改影响接口/状态机/错误码/数据模型 + 头部追加变更记录），不重写 |

## 2. 输出清单

- 文件：`persistent-assets/design/_baseline/02-详细设计.md`（项目级 baseline，固定路径）
- frontmatter：`stage: 02-dld-design` / `scope: <feature-slug 或本次 PRD 主题>` / `version` / `date` / `sources`
- 模板：`../templates/软件详细设计.template.md`
- 模块数较多（≥6）时，可拆为：
  - `02-详细设计.md`（总览 + 索引）
  - `02-详细设计-<模块名>.md`（每模块一份，frontmatter `scope` 填模块名）

## 3. 派生方法（HLD → LLD）

| 章节 | 上游来源 | 派生规则 |
|------|---------|---------|
| 模块概述 | HLD 模块清单 | 一句话职责扩展为 1-2 段，含为什么这个边界、不做什么 |
| 依赖清单 | HLD 模块间依赖 | 上游 + 下游全部列出，标注同步/异步、是否可选、降级路径 |
| 对外接口契约 | HLD 模块通信 + 业务规则 | 每个 public 接口必填：**接口类型（业务接口=领域服务/UseCase/进程内；API 接口=系统服务/IPC/AIDL/HTTP/跨进程或端云边界）**+ 签名 + 前置 + 后置 + 错误码 + 幂等性 + 超时 + 重试语义 |
| 内部行为规约 | 业务规则 + 聚合根不变式 | 把"何时拒绝/何时通过"翻译为可断言条件 |
| 状态机 | 聚合根状态描述 + 业务规则 | 每个状态画清楚 entry/exit 动作、非法转移如何拒绝 |
| 关键时序图 | 业务流程 + 跨模块通信 | 至少：1 happy path + 每个错误码至少 1 条触发路径 |
| 数据模型 | 聚合根 + HLD 存储映射 | DDL/DAO 接口/索引策略；不写完整 SQL，写表结构 + 关键约束 |
| 并发与一致性 | HLD 一致性策略 | 显式声明锁粒度 / 乐观锁版本号 / 事务边界 / 重入语义 |
| 可观测性 | NFR 度量方式 | 每个关键路径列出埋点：日志关键字段 / 指标名 / Trace 标签 |

## 4. 章节结构

完全按 `../templates/软件详细设计.template.md`。模板含：
- 顶层（覆盖整篇）：1-3 节总览与约定
- 模块层（按 HLD 模块清单逐模块复制）：第 4 节为模块详述模板，需为每个模块复制并填空

## 5. 必填字段（每个模块都必填）

- [ ] 模块概述（含"做什么 / 不做什么"）
- [ ] 依赖清单（同步/异步、可选/必选、降级）
- [ ] 对外接口契约表（每个接口必填：**接口类型** + 签名 + 前置 + 后置 + 错误码 + 幂等）
- [ ] 内部行为规约（≥3 条断言级规则）
- [ ] 状态机（Mermaid `stateDiagram-v2`，标 entry/exit/非法转移）
- [ ] 关键时序图（≥1 happy + 关键异常）
- [ ] 数据模型（表结构 + 关键索引 + 约束）
- [ ] 并发与一致性策略
- [ ] 可观测性埋点（≥3 处）
- [ ] 模块级风险与回退（≥1 条）

## 6. 自审 — Contract Completeness（契约完整度）

每个对外接口必须同时给出六要素，缺一不可：

| 要素 | 检查 |
|------|------|
| 接口类型 | 业务接口（领域服务 / UseCase / 进程内调用，单测覆盖）/ API 接口（系统服务 / IPC / AIDL / HTTP / 跨进程或端云边界，集成或契约测试覆盖）必须二选一标注，决定下游测试策略 |
| 签名 | 函数名、入参、返回值、抛异常类型 |
| 前置条件 | 调用前必须满足的不变式 |
| 后置条件 | 调用成功后保证的状态 |
| 错误码/异常 | 全部可能的失败码及触发条件 |
| 幂等性 | 重复调用是否安全；如非幂等需说明边界 |

附加：

- 状态机是否覆盖了业务领域所有业务规则中提到的状态？
- 状态机是否每个非法转移都显式拒绝（而不是默默忽略）？
- 时序图是否覆盖了接口契约里全部错误码的触发路径？
- 数据模型字段是否对应到聚合根中的实体/值对象？

## 7. 常见踩坑

| 踩坑 | 表现 | 纠偏 |
|------|------|------|
| 接口契约只写签名 | 没写前置/后置/错误码/幂等 | 一律补齐六要素，缺哪个补哪个 |
| 状态机只画 happy 转移 | 缺非法转移的拒绝逻辑 | 每状态枚举入栈事件，标注允许/拒绝及拒绝行为 |
| 错误码不闭环 | 接口列了错误码但时序图没有触发路径 | 反向校验：每个错误码至少 1 条路径 |
| 时序图全 happy | 没有异常分支 | happy 之外，关键异常至少 1 张图（鉴权失败/超时/降级触发等） |
| 并发策略含糊 | "用锁" / "用事务" | 写清：什么粒度锁、悲观/乐观、事务边界精确到方法 |
| 把 ORM 实体当数据模型 | 直接拷代码片段 | 写表结构（字段+类型+约束），关注**契约**而非**实现** |
| 模块独立性丢失 | 模块 A 章节里频繁引用模块 B 内部细节 | 只通过对外接口引用 B，不暴露 B 的内部 |
