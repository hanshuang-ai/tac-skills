## 6. 自审检查

通用 4 类 + Stage 4 专属第 5 类：

**第 5 类 — Coverage（覆盖完整度）：**

按"集合相等"原则反向校验：

| 集合 A（输入） | 集合 B（验收用例） | 校验 |
|---------------|------------------|------|
| PRD 全部 in-scope 项 | 验收范围 + 功能用例 | 一一对应（追溯表） |
| DLD 每个 public 接口 | 功能用例 happy 路径 | 接口 ⊆ happy 用例的"被测对象" |
| DLD 全部错误码 | 异常用例 | 错误码 ⊆ 异常用例的"期望错误码" |
| DLD 每个状态机的非法转移 | 状态机拒绝用例 | 转移 ⊆ 拒绝用例的"被测转移" |
| Stage 1 每条业务规则 | 业务规则用例 | R# ⊆ 用例标签 |
| HLD 每条 NFR | 性能/安全/可观测用例 | NFR ⊆ 用例覆盖标签 |

任一行不满足"⊆"即自审失败，必须补用例（不要降低标准、不要假装覆盖）。

附加自审：
- 每条用例的"通过判据"是否量化？（"成功"不是判据，"返回 200 且 body.code=0 且耗时<200ms"才是）
- 每条用例的"执行方式"是否落地？（"自动化"不够，要写"通过 `./gradlew testDebugUnitTest` 跑 X 测试类"）

## 8. 常见踩坑

| 踩坑 | 表现 | 纠偏 |
|------|------|------|
| 用例描述像功能说明 | "支持登录" | 改成 Given-When-Then：Given 用户已注册 / When 输入正确密码 / Then 返回 token 且过期 24h |
| 通过判据模糊 | "正常工作" / "无明显错误" | 写量化判据：响应码、字段值、耗时、日志中包含的关键字段 |
| 错误码漏覆盖 | DLD 列了 10 个错误码，验收只覆盖 5 个 | 补齐；每个错误码至少 1 条触发用例 |
| 性能验收无负载 | "P99 ≤ 200ms" | 必须给负载：在 X QPS 下、Y 并发下、Z 数据量下 |
| 安全验收形式化 | "做了鉴权" | 写出测试方法：未授权请求返回 401 / 越权访问返回 403 / 注入攻击被过滤 |
| 回归基线缺失 | "无" | 实在无历史 issue 也要显式写出"无历史回归基线，本 feature 为首次实现"，并留出后续追加锚点 |
| 执行方式空写 | "自动化测试" | 写到具体类/命令/脚本，否则 implementation 阶段会重新发明 |
