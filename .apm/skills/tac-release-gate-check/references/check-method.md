# 准出审核方法

## 审核原则

准出审核是证据审计，不是主观评价。

每一条适用准出标准都要能回答：

| 问题 | 要求 |
|---|---|
| 标准是否适用？ | 适用则检查，不适用必须说明原因 |
| 证据是什么？ | 命令、测试、CI、代码路径、文档段落、审批记录 |
| 证据是否足够？ | 不足则 BLOCKED，不得 PASS |
| 风险是否阻塞？ | 硬性标准失败或缺证据即不准出 |

## 证据优先级

从高到低：

1. 本地或 CI 命令结果：构建、测试、lint、静态扫描、覆盖率。
2. 测试报告：单测、集成测试、UI 自动化、手动验收记录。
3. 代码和配置：实现路径、差异、配置、发布产物信息。
4. 设计/需求文档：验收条件、接口契约、发布策略。
5. 人工审批：风险接受、发布审批、安全/架构确认。

## 常见 BLOCKED 情况

- 标准要求测试通过，但没有测试结果。
- 标准要求回滚方案，但发布材料没有回滚路径。
- 标准要求兼容旧版本，但没有升级/迁移验证证据。
- 标准要求缺陷门槛，但没有缺陷清单或风险确认。
- 标准要求安全扫描，但没有扫描报告或审批。

## 常见 N/A 情况

- 标准仅适用于版本发布，本次只是本地需求开发。
- 标准仅适用于数据迁移，本次没有存储结构和历史数据影响。
- 标准仅适用于 UI 变更，本次只改内部脚本或文档。

N/A 必须说明为什么不适用。
