## 目标

生成**可执行的软件准出标准**，作为后续编码完成、版本发布或系统级变更准出的判断依据。

本 Skill 只负责“定规则”，不负责判断当前实现是否通过。编码后准出判断使用 `tac-release-gate-check`。

## 输出

默认落盘：

```text
persistent-assets/design/_baseline/04-准出标准.md
```

准出标准必须包含三级：

| 层级 | 作用 | 典型问题 |
|---|---|---|
| 项目/系统级 | 长期质量底线和架构/业务约束 | 这个系统必须长期守住什么底线？ |
| 版本发布级 | 本次版本能否上线 | 这个版本能不能发？ |
| 需求/功能级 | 单个需求或代码变更是否完成 | 这个需求算不算完成？ |

## 边界

- 本 Skill 生成或维护准出标准文档。
- 不执行最终准出结论。
- 不把泛化 QA checklist 当作准出标准。
- 已吸收原质量交付 checklist 的质量维度能力；对外不再保留独立质量 checklist 入口。
- 不替代 `tac-commit-review` 内置的提交级质量门禁。

## 输入来源

按可获得性读取，不因缺失自动阻塞，但必须把缺失来源写入“假设与待确认问题”。

| 来源 | 用途 |
|---|---|
| `00-业务与领域设计.md` | 业务规则、领域对象、权限、状态流转、业务不变量 |
| `01-概要设计.md` | 架构边界、模块职责、NFR、依赖、数据流、发布影响 |
| `02-详细设计.md` | 接口契约、状态机、异常处理、存储、兼容、回滚策略 |
| PRD / issue / 验收标准 | in-scope 范围、用户价值、验收条件 |
| 实现代码 | 实际模块边界、测试策略、可观测性和兼容性落点 |
| 宪章 / governance 文档 | 项目级硬性约束和质量底线 |
| CI / 构建 / 测试配置 | 可作为后续检查所需证据类型 |

质量维度派生参考 `references/quality-dimensions.md`。生成准出标准时，至少检查这些维度是否需要转化为硬性标准或建议检查项：功能完成度、性能要求、代码质量、安全准出、架构合理性、可观测性、兼容性。

## Step 0: 收集范围

至少明确：

| 字段 | 必填 | 说明 |
|---|---|---|
| 交付主题 | 是 | feature-slug、需求名、版本号或发布主题 |
| 代码目标 | 否 | branch、commit、tag、模块路径或变更范围 |
| 标准文件路径 | 否 | 默认 `persistent-assets/design/_baseline/04-准出标准.md` |
| 上游设计文档 | 否 | 默认探测 baseline 设计文档 |
| 准出层级 | 否 | 默认三级都生成 |

如果信息不足但可以从仓库探测，先探测；仍不足时用 OPEN 问题记录，不要编造标准来源。

## Step 1: 派生准出标准

1. 读取 `references/derivation-method.md`。
2. 读取 `references/quality-dimensions.md`，把适用的质量维度转化为三级准出标准。
3. 汇总输入来源，按三级准出层级提取风险和约束。
4. 使用 `templates/准出标准.template.md` 生成或更新标准文件。
5. 每条标准必须包含：ID、标准、证据、来源、负责人、状态。
6. 区分“硬性准出标准”和“建议检查项”。
7. 运行 `scripts/validate_release_gate.py <标准文件>` 做结构校验。
8. 输出变更摘要和待确认问题，不执行 git。

## 标准写法

优先使用可判定表达：

- 好：`P0/P1 缺陷为 0；P2 缺陷均有 owner、规避方案和发布审批记录。`
- 好：`从上一生产版本升级后，用户本地数据保留，且有自动化或记录化手动验证证据。`
- 好：`新增公开接口的成功、失败、权限拒绝三类路径均有测试或验收证据。`
- 坏：`测试充分。`
- 坏：`性能正常。`
- 坏：`代码质量较好。`

## 硬规则

1. **准出标准必须分三级**：项目/系统级、版本发布级、需求/功能级。
2. **硬性标准必须可判定**：避免“质量良好”“测试充分”等不可验证表述。
3. **标准必须可追溯**：来源可以是需求、设计、代码、宪章、测试策略或明确推理。
4. **只读业务代码**：本 Skill 可生成/更新准出标准文档，不修改业务实现。
5. **不执行 git**：不 add、不 commit、不打 tag。
