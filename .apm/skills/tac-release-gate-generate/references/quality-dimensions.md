# 质量维度到准出标准的映射

生成准出标准时，把以下 7 个质量维度作为候选来源。不是每个维度都必须成为硬性标准，但每个适用维度都必须被显式处理：写入硬性标准、写入建议检查项，或在“假设与待确认问题”中说明缺失依据。

| Code | 维度 | 准出关注点 |
|---|---|---|
| FN | 功能完成度 | PRD in-scope、业务规则、接口契约、状态机、异常路径是否落地 |
| PF | 性能要求 | 关键路径耗时、吞吐、资源占用、启动、容量边界是否满足 NFR |
| CQ | 代码质量 | 测试覆盖、复杂度、静态扫描、Review、重复代码、规范遵从 |
| SC | 安全准出 | 鉴权、输入校验、数据保护、依赖漏洞、权限最小化、隐私合规 |
| AR | 架构合理性 | 模块边界、跨上下文调用、跨切面策略、一致性策略、技术栈一致 |
| OB | 可观测性 | 关键日志、指标、Trace、告警、错误定位字段是否可用 |
| CP | 兼容性 | 系统/设备矩阵、协议兼容、历史数据迁移、国际化、无障碍 |

## 派生规则

### 功能完成度

从 PRD、业务领域设计、LLD 和实现代码派生：

- in-scope 条目都有实现入口或模块对位。
- 业务规则在代码中有校验、状态约束或等价实现。
- 对外接口契约、错误码、幂等、前置/后置条件可验证。
- 非法状态转移、异常路径、降级路径有显式处理。
- 历史已修复缺陷有回归验证要求。

### 性能要求

从 HLD NFR、部署拓扑、容量约束和代码路径派生：

- 关键路径 P50/P95/P99 或启动耗时有阈值和度量方式。
- 峰值内存、CPU、网络、磁盘 IO 有上限或观察指标。
- 并发、吞吐、容量边界有测试条件和通过标准。
- 缺少上游阈值时，记录 OPEN 问题，不编造数字。

### 代码质量

从项目宪章、编码规范、CI、测试配置和实现代码派生：

- 新增/修改代码有单测、集成测试或明确验收证据。
- 静态扫描、lint、编译、测试命令作为准出证据。
- 复杂度、函数长度、重复代码等阈值优先使用项目规范。
- 关键 PR 或核心模块变更需要 Review 证据。

### 安全准出

从 HLD 安全策略、LLD 接口契约、Manifest/权限、依赖配置和合规要求派生：

- public 接口鉴权/授权路径完整。
- 外部输入有边界校验，敏感数据加密、脱敏或不落盘。
- 依赖漏洞、第三方 license、运行时权限有准出要求。
- 涉及个人信息时，用户同意、收集范围、日志脱敏必须可验证。

### 架构合理性

从 HLD 模块清单、上下文映射、跨切面策略和实际依赖派生：

- 实现模块边界和依赖方向符合 HLD。
- 跨限界上下文调用通过声明的接口或防腐层。
- 日志、监控、鉴权、事务、限流、安全等跨切面策略落地。
- 技术栈或部署拓扑偏离设计时，需要决策记录或风险接受。

### 可观测性

从 HLD/LLD 可观测性设计、日志策略、监控配置和代码派生：

- 关键路径日志包含 traceId、错误码和必要业务上下文。
- 指标命名、标签、采集方式和查询入口明确。
- 关键告警有阈值、响应人或发布观察要求。
- 日志不得泄露敏感信息。

### 兼容性

从 PRD 兼容矩阵、Android minSdk/API、LLD 数据模型和上下游协议派生：

- 系统版本、设备、分辨率或渠道矩阵有验证要求。
- 协议向前/向后兼容和版本协商可验证。
- 历史数据可读，schema 迁移有回滚或恢复策略。
- 有国际化/无障碍要求时，关键流程必须纳入准出标准。

## 覆盖自审

生成后反向检查：

| 上游集合 | 准出标准覆盖 |
|---|---|
| PRD in-scope 全集 | 功能级标准 |
| HLD NFR 全集 | 系统级或版本级性能标准 |
| HLD 模块清单 | 系统级架构标准 |
| LLD 接口契约/错误码 | 功能级接口和异常标准 |
| 项目宪章质量门类 | 对应层级的硬性标准 |
| 历史 issue / lessons | 回归或风险标准 |

任一集合未覆盖时，补标准，或在“假设与待确认问题”中说明为何本次不适用。
