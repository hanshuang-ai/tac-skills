# 软件概要设计 — 派生方法

## 1. 输入清单

| 来源 | 文件 | 用法 |
|------|------|------|
| 必读 | `persistent-assets/design/_baseline/00-业务与领域设计.md` | 限界上下文 → 模块映射、领域事件 → 跨模块通信 |
| 必读 | 用户在 Step 0 提供的 PRD | 非功能需求、性能/容量约束 |
| 必读 | 项目入口文件（`PROJECT.md` / `AGENTS.md` / `CLAUDE.md` / `README.md` 之一）「场景 → 必读文档」映射 | 当前项目技术栈与既有架构约束；未命中则按通用 Android/Kotlin 最佳实践派生（默认 Jetpack 生态） |

| 维护模式必读 | 上一版 `01-概要设计.md` + 变化清单 | 增量修改（仅改影响章节 + 头部追加变更记录），不重写 |

## 2. 输出清单

- 文件：`persistent-assets/design/_baseline/01-概要设计.md`（项目级 baseline，固定路径）
- frontmatter：`stage: 01-hld-design` / `scope: <feature-slug 或本次 PRD 主题>` / `version` / `date` / `sources`
- 模板：`../templates/软件概要设计.template.md`

## 3. 派生方法（业务领域 → 概要设计）

| 章节 | 上游来源 | 派生规则 |
|------|---------|---------|
| 技术栈选型 | PRD 非功能需求 + 项目现状 | 列 2-3 备选方案，每个方案给"满足约束/不满足"对照表，推荐方案带理由 |
| 架构风格 | 限界上下文规模 + 团队规模 + 部署约束 | 单体/分层 / 模块化单体 / 微服务等，必须给出"为什么不是其他风格"的反驳 |
| 模块清单 | 限界上下文清单 | 每个限界上下文映射到 1 个或多个模块；多对一禁止（合并即跨上下文） |
| 模块间依赖 | 上下文映射图 | 把领域事件/命令翻译为同步调用 / 异步消息 / 共享存储 |
| 外部依赖 | PRD 外部系统列表 + 既有平台 | 协议、版本、可用性 SLA、降级策略必须填齐 |
| 跨切面策略 | 项目通用规范 | 日志/监控/鉴权/事务/限流/安全 至少各 1 段；不适用要写"不适用：原因" |
| 数据存储与一致性 | 聚合根 + 业务规则 | 聚合根 → 表/集合的映射策略；跨聚合一致性（强一致/最终一致）必须显式 |
| 部署拓扑 | 模块清单 + 非功能需求 | 进程边界、容器/服务边界、扩缩容粒度 |
| 非功能达成路径 | PRD 非功能 + 上述设计 | 每条 NFR 写"通过什么设计达成 + 如何度量"，不能只写"满足" |

## 4. 章节结构

完全按 `../templates/软件概要设计.template.md`。

## 5. 必填字段

- [ ] 技术栈选型表（≥2 备选 + 推荐 + 推荐理由）
- [ ] 架构风格选择 + 反驳替代风格（≥1 段）
- [ ] C4 Context 图（Mermaid `graph` 或 `flowchart`）
- [ ] C4 Container 图（同上，颗粒度到模块）
- [ ] 模块清单表（每行：模块名 / 一句话职责 / 对应限界上下文 / 主要依赖）
- [ ] 模块间依赖图 + 通信方式标注
- [ ] 外部依赖清单表（系统名 / 协议 / 版本 / SLA / 降级策略）
- [ ] 跨切面策略 6 条（日志/监控/鉴权/事务/限流/安全），每条 1-3 句话或显式 "不适用：原因"
- [ ] 数据存储与一致性策略
- [ ] 部署拓扑图
- [ ] NFR 达成路径表（覆盖 PRD 全部非功能需求）
- [ ] 关键风险与应对（≥3 条）

## 6. 常见踩坑

| 踩坑 | 表现 | 纠偏 |
|------|------|------|
| 技术栈"凑齐"备选 | 写 3 个明显不合适的方案陪跑 | 备选必须是真实候选，理由互斥；只有 1 个方案合理就只写 1 个并说明无替代 |
| 模块跨上下文 | 一个模块同时处理订单和库存 | 拆分；若必须共存（如 saga 协调器），单独标注角色 |
| 跨切面写得太抽象 | "采用统一日志框架" | 写出框架名 + 关键 logger 命名规范 + 错误日志结构 |
| 一致性策略含糊 | "保证数据一致" | 显式声明强一致（事务边界）/ 最终一致（补偿/对账机制）/ 弱一致（接受窗口） |
| NFR 复制 PRD 不展开 | 只列 NFR 不写达成路径 | 每条 NFR 必须配"通过什么设计达成 + 如何度量" |
| 跳过反驳 | 只说选了 X | 选 X 必带"为什么不是 Y / Z"，否则未来无法回看决策 |
