## Step 1: 按 Feature 需求模板收集输入（不可跳过）

检查用户输入是否覆盖以下七个字段。**缺失任何必填项必须追问，不得跳过直接执行。**

| 字段 | 必填 | 对应模板 | 说明 |
|------|------|---------|------|
| 功能 | 是 | 【功能】 | 一句话描述功能目标 |
| 场景 | 是 | 【场景】 | 用户操作路径，从哪里进入、做什么操作、看到什么结果 |
| 输入 | 是 | 【输入】 | 接收什么数据，数据来源 |
| 输出 | 是 | 【输出】 | 产出什么结果，如何展示 |
| 约束 | 是 | 【约束】 | 技术栈/性能/兼容性/安全/车机场景要求 |
| 验收标准 | 是 | 【验收标准】 | 验收用例编号 / 设计文档路径+章节 / 内联 Given-When-Then / 内联通用判据 / 复用既有测试基线（详细规则见 Step 1.5） |
| 参考 | 否 | 【参考】 | 相关设计文档、API 文档、截图路径 |

**追问模板（缺失时输出完整模板要求用户补充）：**

> 【输出】：<产出什么结果，如何展示>

> 【验收标准】：<AC 编号列表 / 设计文档路径+章节 / 内联 Given-When-Then / 内联通用判据 / 复用既有测试基线>
> 【参考】：<相关设计文档、API 文档、截图路径>（可选）

**所有必填字段收集完成后必须执行 Step 1.5 质量校验，校验通过才能进入 Step 2。**

## Step 1.5: 输入质量校验（不可跳过）

对七字段逐项执行以下校验；任意一项失败必须重新追问对应字段，**不得进入 Step 2**。

**通用校验（适用于所有必填字段）：**

1. **非空校验**：字段值不为空字符串、不为单个空白
2. **占位符校验**：字段值不含 `TODO` / `TBD` / `待定` / `稍后补充` / `暂略` / `见文档`（无具体路径引用）等占位语
3. **意图清晰**：单一字段不含 2 种以上互相矛盾的解读

**【验收标准】字段额外校验（5 选 1，与 `.specify/memory/input-quality-constitution.md §6` 对齐；本仓库无该宪章时按下表自治执行）：**

| 形式 | 合规示例 | 校验规则 |
|------|---------|---------|
| AC 编号列表 | `AC-HOM-FN-01..05, AC-DL-RU-12` | 至少 1 个匹配 `AC-[A-Z]+-[A-Z]+-\d+` 的编号 |
| 设计文档路径+章节 | `persistent-assets/design/<slug>/软件验收标准.md §3.2` | 路径可读 + 章节锚点（`§数字(.数字)*` / `#anchor` / 行号） |
| 内联 Given-When-Then | `Given 已登录用户在首页 / When 点击下载按钮 / Then 5 秒内进入下载列表` | 至少 1 组完整三元组 |
| 内联通用判据 | `调用方应能取消下载并释放磁盘空间，UI 不残留 loading 态` | 至少 1 条含具体动作 + 可观察期望结果 |
| 复用既有测试基线 | `无独立验收文档；复用 feature-download/.../DownloadCoordinatorTest 既有测试基线` | 显式声明"无独立验收文档" + 至少 1 个现有测试文件路径（路径可读） |

**典型不合规（须追问）：** 空字符串、单纯 "无"、单纯 "见验收标准文档"、`TODO` / `TBD` / `待定` 等占位语、AC 编号格式错误、Given/When/Then 残缺、通用判据无具体动作或无可观察结果。

**追问模板（字段层校验失败时）：**

> 【验收标准】字段当前内容 `<具体内容>` 不满足质量约束（`<失败原因>`）。请按以下任一形式补充：
> - AC 编号列表（如 `AC-HOM-FN-01..05`）
> - 设计文档路径+章节（如 `persistent-assets/design/<slug>/软件验收标准.md §3.2`）
> - 内联 Given/When/Then 至少 1 组
> - 内联通用判据至少 1 条（含具体动作与期望结果）
> - 显式"无独立验收文档"+复用的既有测试文件路径

## Step 2: 确认宪章适用范围

打开仓库根目录 `PROJECT.md` 的「场景与必读文档映射」章节，按本次变更命中的场景列出对应的专项规范与相关文档清单（同一任务命中多个场景须并集，不得只取其一）。

将清单提交用户确认后再进入 Step 3，**不得在本 Skill 内固定列举专项规范条目**——映射以 `PROJECT.md` 当前内容为准，避免与单一文档真理源出现漂移。

## Step 3: 启动 Speckit 流程

将收集到的结构化输入作为参数，执行 `/speckit-specify`。

后续按 Speckit 标准流程继续：
specify → clarify → plan → tasks → implement
