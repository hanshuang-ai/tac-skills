## 何时用 / 何时不用  111111111111111111111

**用本 skill**：

- 用户要求修复缺陷、异常、崩溃、卡顿、显示异常、行为不符合预期等问题
- 用户给出测试失败、Maestro 失败、QA 反馈、线上反馈，希望定位并修复
- 已确认目标是修复现有行为，而不是开发新功能或重写需求

**不要用**（引导到正确入口）：

- 新增功能、行为调整、重构、适配、迁移 → 使用 `tac-feature`
- 只做经验复盘或流程沉淀 → 使用 `tac-experience-distill`
- 只跑提交级质量门禁或提交前检查 → 使用 `tac-commit-review`
- 创建或维护缺陷台账、BUG 文件、索引文件 → 交给外部缺陷系统或后续专门流程，本 skill 不处理

## 核心约束

1. **最小输入只有三项**：现象、预期、复现步骤。缺任一项必须追问，不得进入定位和改代码。
2. **按问题类型动态追问证据**：日志、截图、复现率、环境、版本窗口等不是通用必填，但命中对应问题类型时必须补齐。
3. **先复现再修复**：必须先建立失败证据（单测、仪器测试、Maestro、手动复现说明之一），不得凭猜测直接修改。
4. **修根因，不修表象**：修复前必须写出根因证据链和已排除假设。
5. **本 skill 不维护 BUG 文件**：不创建 `BUG*.md`、不维护 `bugs.yaml` / index、不中途写 `issues_and_learnings.md`、不执行 `git add` / `git commit`。

## Step 1：收集最小输入（不可跳过）

**做什么**：检查用户输入是否包含以下三项。三项齐全后，先结构化复述给用户；缺失时只追问缺失项。

| 字段 | 必填 | 说明 |
|---|---|---|
| 现象 | 是 | 什么操作后出现什么异常结果 |
| 预期 | 是 | 正常情况下应该是什么结果 |
| 复现步骤 | 是 | 可按 1-2-3 执行的问题路径 |

**追问模板（缺失时使用）**：

> 为了准确定位和修复问题，请补充缺失信息：
>
> 【现象】：什么操作后出现什么异常结果？
> 【预期】：正常情况应该是什么？
> 【复现步骤】：操作路径 1-2-3？

**产物**：一段结构化输入摘要：

```text
现象：...
预期：...
复现步骤：1. ... 2. ... 3. ...
```

**验证**：现象、预期、复现步骤任一为空、含糊到无法判断、或相互矛盾时，继续追问，不进入 Step 2。

## Step 2：按问题类型补充诊断信息

**做什么**：根据 Step 1 的问题类型判断是否需要追加证据。只追问对定位必要的信息，避免把所有字段都变成入口门槛。

| 问题类型 | 必须补充 | 目的 |
|---|---|---|
| Crash / ANR / 闪退 / 无响应 | 关键日志、堆栈、logcat、bugreport 路径之一 | 确认崩溃点和线程状态 |
| UI 显示异常 / 错位 / 重叠 / 不显示 | 截图或录屏，页面路径 | 对齐实际视觉问题 |
| 偶现 / 卡顿 / 卡住 | 复现率、触发条件、最近一次出现时间 | 区分稳定缺陷和环境噪声 |
| 回归问题 | 最近正常版本、异常版本、疑似变更窗口 | 缩小排查范围 |
| 接口 / 数据相关 | 请求条件、响应摘要、账号或数据前置条件（敏感信息脱敏） | 排除数据和环境差异 |
| 特定设备 / 车型 / 系统版本问题 | 设备型号、系统版本、构建类型 | 避免在错误环境修复 |

**产物**：补充一段“诊断信息摘要”。如果没有命中条件必填项，明确写“暂无额外诊断信息要求”。

**验证**：命中条件必填但用户未提供时，停止并追问；不得用“稍后补”继续定位。

## Step 3：分诊

**做什么**：在改代码前判断问题性质，避免把需求歧义、环境问题、数据问题误修成代码变更。

| 类型 | 判断 | 下一步 |
|---|---|---|
| 代码缺陷 | 行为与明确预期不符，且可通过测试或复现证据证明 | 进入 Step 4 |
| 需求歧义 | 预期不明确，或不同角色期望冲突 | 停止，请用户确认预期 |
| 环境 / 数据问题 | 依赖设备、配置、账号、服务返回或脏数据 | 先补环境 / 数据证据 |
| 不可复现 | 按步骤无法复现，且缺少日志 / 截图 / 录屏等证据 | 停止，要求补证据 |
| 重复问题 | 已有同类修复或已知限制 | 引导用户确认是否继续处理 |

**产物**：一句分诊结论，例如：

```text
分诊：代码缺陷。依据：复现步骤稳定触发，预期明确，日志指向下载状态转换。
```

**验证**：分诊不是“代码缺陷”时，不进入 Step 4；先解决分诊阻塞。

## Step 4：确认规范适用范围

**做什么**：打开仓库根目录 `PROJECT.md` 的「场景与必读文档映射」章节，按本次 Bug 命中的场景列出对应专项规范与相关文档清单。同一任务命中多个场景时取并集。

**产物**：向用户列出将读取 / 遵循的规范清单。

**验证**：用户确认前不得进入代码定位。不得在本 skill 内固定列举专项规范条目，映射以 `PROJECT.md` 当前内容为准。

## Step 5：定位入口代码

**做什么**：

1. 根据现象、复现步骤、诊断信息搜索相关代码
2. 定位入口路径、状态流、数据流或 UI 渲染链路
3. 阅读相关实现和已有测试，理解当前行为

**产物**：

```text
入口代码：path/to/File.kt#function
相关测试：path/to/FileTest.kt 或 无
当前逻辑摘要：...
```

**验证**：不能只列文件名；必须说明为什么这些文件与现象相关。

## Step 6：Red — 建立失败证据

**做什么**：优先写或更新一个失败测试来精确复现 Bug。测试层级按问题类型选择：

| 场景 | 优先证据 |
|---|---|
| 纯逻辑 / 状态转换 / 数据映射 | 单元测试 |
| UI 行为 / 页面跳转 / 交互流程 | 仪器测试或 Maestro |
| Crash / ANR | 最小复现测试 + 日志证据 |
| 无法自动化 | 手动回归步骤 + 无法自动化原因 |

示例命令：

```bash
./gradlew testDebugUnitTest --tests "*.XxxTest"
```

**产物**：失败证据记录：

```text
复现方式：单元测试 / 仪器测试 / Maestro / 手动
命令：...
结果：失败
失败原因：与 Bug 现象一致 / 不一致
```

**验证**：失败必须与 Bug 描述一致。测试已通过、失败原因不一致、或只是编译错误时，重写复现证据，不进入 Step 7。

## Step 7：根因分析

**做什么**：从失败证据向上追溯，形成可审查的根因链路。不要把第一个可改点当根因。

| 项 | 内容 |
|---|---|
| 直接失败 | 测试或复现中观察到的失败 |
| 触发条件 | 哪个输入、状态或事件触发 |
| 已排除假设 | 至少列出已检查但不是根因的方向 |
| 根因位置 | 文件、函数、状态流或数据流 |
| 同类扫描 | 同模块是否存在相同模式 |

**产物**：根因分析结论。

**验证**：未明确根因位置、未说明证据、或只描述“可能是”时，不进入 Step 8。

## Step 8：Green — 最小化修复

**做什么**：

1. 只修改根因相关代码
2. 不顺手重构无关模块
3. 不扩大功能行为
4. 运行 Step 6 的复现证据，确认从 Red 变 Green

示例命令：

```bash
./gradlew testDebugUnitTest --tests "*.XxxTest"
```

**产物**：修复摘要：

```text
修改文件：...
修复点：...
为什么是最小修复：...
```

**验证**：复现证据未变 Green 时继续修复；不得跳到 Step 9。

## Step 9：Refactor 与回归验证

**做什么**：

1. 在测试保护下清理修复代码
2. 补充必要边界测试
3. 运行局部测试和项目约定的回归命令

示例命令：

```bash
./gradlew testDebugUnitTest
./gradlew assembleDebug
```

**产物**：验证摘要：

```text
局部复现验证：PASS
单元测试：PASS / 未运行（原因）
编译验证：PASS / 未运行（原因）
手动或设备回归：PASS / 未运行（原因）
```

**验证**：不能只报告“已修复”；必须列出实际执行过的命令或未执行原因。

## Step 10：收尾

**做什么**：向用户输出最终修复结论，不做缺陷台账维护。

必须包含：

- 现象 / 预期 / 复现步骤摘要
- 分诊结论
- 根因
- 修复点
- 验证命令与结果
- 是否建议后续调用 `tac-experience-distill`

**沉淀建议规则**：

- 同类问题重复出现
- Critical / Blocker 级问题
- AI 排查中出现明显误判
- 暴露项目规范、测试策略或流程缺口

命中以上任一项时，只建议用户后续调用 `tac-experience-distill`；本 skill 不直接写经验沉淀文件。

## Stop conditions

- 现象 / 预期 / 复现步骤任一缺失时停止，按追问模板补齐
- 条件必填证据缺失时停止，例如 crash 无日志、UI 问题无截图、偶现无复现率
- 用户未确认规范适用范围清单时停止，不进代码定位
- 分诊不是代码缺陷时停止，先澄清预期、环境或数据
- 复现证据无法 Red，或 Red 原因与 Bug 不一致时停止，重写复现证据
- 未明确根因就准备改代码时停止
- 修复范围超出根因相关代码时停止，拆到其他 feature / refactor 流程
- 本 skill 不执行 `git add` / `git commit`

## 常见问题

| 现象 | 处置 |
|---|---|
| 用户只说“修一下 X 不好用” | 追问现象、预期、复现步骤三项 |
| 用户不给环境信息 | 若不命中条件必填，不阻塞；若是设备 / 回归 / 数据问题，按 Step 2 追问 |
| Crash 只有一句“闪退” | 必须追问日志、堆栈、logcat 或 bugreport 路径 |
| UI 问题没有截图 | 必须追问截图 / 录屏和页面路径 |
| 测试写完就是 Green | 测试没有复现 Bug，重写到 Red |
| 失败原因是编译错误 | 先修测试自身错误，再确认 Red 是否与 Bug 一致 |
| 找到的“根因”只是表象 | 继续向上追溯，补已排除假设和证据 |
| 修复后想顺手重构 | 只允许 Step 9 中在测试保护下做局部清理；大重构走单独流程 |
| 高频问题需要沉淀 | 收尾时建议调用 `tac-experience-distill`，本 skill 不直接写沉淀文件 |
