# 跨阶段一致性检查

每进入下一阶段前，执行对应小节的检查。**全部通过才能开始派生**；任一不通过必须先回上一阶段补全（重审）。

## 1. 进入 Stage 1 前

仅检查 Step 0 输入（首次模式）：

- [ ] PRD 文件存在且可读
- [ ] feature-slug 已确认（kebab-case，非通用名；用于 frontmatter `scope` 字段与下游交棒，不再用于目录命名）
- [ ] 范围声明 in/out 各 ≥ 3 项
- [ ] 项目级 baseline 目录 `persistent-assets/design/_baseline/` 已存在（由 `tcli init` 创建；首次运行若不存在需先 `mkdir -p`）

### 1.M 维护模式额外检查（如 Step 0 提供「迭代来源」字段）

- [ ] 项目级 baseline 目录 `persistent-assets/design/_baseline/` 下 4 份产物齐全（`00-业务与领域设计.md` / `01-概要设计.md` / `02-详细设计.md` / `03-准出标准设计.md`）
- [ ] 迭代来源 commit hash 在仓库 git history 中存在（`git cat-file -e <hash>` 不报错），可用于比对旧版本
- [ ] 变化清单按类型分组（澄清 / 修改 / 新增 / 废弃 / 重命名），每条引用 PRD 章节号
- [ ] 变化清单的每条变化已经过用户确认（不是 LLM 自行猜测）
- [ ] 维护策略已与用户确认（默认就地增量修订；如需要保留旧版本对照请走 git history / branch / tag，不再通过新建目录复制实现）
- [ ] M1 影响矩阵已输出给用户并审阅通过（含每条变化对 4 份 baseline 文件各章节的影响）
- [ ] 维护模式下不再执行 Step 1~4 全集生成；改走 Step M 局部修改流程

## 2. 进入 Stage 2 前（Stage 1 完整性 + 派生准备）

### 2.1 Stage 1 自审已通过 + 用户审批

- [ ] `00-业务与领域设计.md` 已落盘到 `persistent-assets/design/_baseline/`（commit 由用户自行决定，不作为进入下阶段的硬门槛）
- [ ] 用户回复确认（"通过/确认/继续"或等价）

### 2.2 Stage 1 内部完整性反查

- [ ] 限界上下文表 ≥ 1 行，每行包含一句话语言模型
- [ ] 上下文映射图存在且每条边标注关系类型
- [ ] 聚合根 ≥ 1 个，每个聚合根含不变式
- [ ] 领域事件 ≥ 3 个，全部过去式命名
- [ ] 业务规则 ≥ 5 条，全部为可断言命题
- [ ] Ubiquitous Language 覆盖全部聚合根/实体
- [ ] 派生与追溯表存在，PRD 元素一一对应

### 2.3 派生准备

- [ ] 已读项目入口文件（`PROJECT.md` / `AGENTS.md` / `CLAUDE.md` / `GEMINI.md` / `README.md` 之一，按 SKILL.md "与项目治理的对接" §1 检测）中本 PRD 命中场景的专项规范清单；未命中则按通用 Android/Kotlin 最佳实践处理
- [ ] 已识别项目既有技术栈与架构约束（按检测顺序：入口文件 Active Technologies 段 / `gradle/libs.versions.toml` / `package.json` 等；避免重复造轮子）

## 3. 进入 Stage 3 前（Stage 2 完整性 + Stage 1→2 派生一致性）

### 3.1 Stage 2 自审已通过 + 用户审批

- [ ] `01-概要设计.md` 已落盘到 `persistent-assets/design/_baseline/`（commit 由用户自行决定，不作为进入下阶段的硬门槛）
- [ ] 用户回复确认

### 3.2 Stage 2 内部完整性反查

- [ ] 技术栈选型 ≥ 2 备选 + 推荐 + 反驳替代
- [ ] 架构风格 + 反驳替代风格段落存在
- [ ] C4 Context + Container 双图存在
- [ ] 模块清单 ≥ 1 行，每行五字段填齐（模块/职责/上下文/上游/下游）
- [ ] 模块间通信表全部标注通信方式 + 失败处理
- [ ] 外部依赖表全部填齐五要素（系统/协议/版本/SLA/降级）
- [ ] 跨切面 6 项全部回应（具体策略或"不适用：原因"）
- [ ] NFR 达成路径表覆盖 PRD 全部非功能需求

### 3.3 Stage 1 → 2 派生一致性

- [ ] **限界上下文 → 模块**：Stage 1 每个 BC 在 Stage 2 模块清单中有 ≥ 1 个对应模块
- [ ] **聚合根 → 存储**：Stage 1 每个聚合根在 Stage 2 §9.1 表中有存储映射
- [ ] **领域事件 → 通信**：Stage 1 每个领域事件在 Stage 2 §6 模块间通信表中能找到对应行（同步/异步/共享存储）
- [ ] **业务规则 → 一致性策略**：Stage 1 涉及跨聚合的业务规则在 Stage 2 §9.2 中有显式一致性级别声明
- [ ] **术语一致**：Stage 2 模块/存储命名沿用 Stage 1 Ubiquitous Language（不引入新业务术语）

任一项 ✗ → 回 Stage 1 补全或回 Stage 2 修订（看缺口在哪一侧）。

## 4. 进入 Stage 4 前（Stage 3 完整性 + Stage 2→3 派生一致性）

### 4.1 Stage 3 自审已通过 + 用户审批

- [ ] `02-详细设计.md` 已落盘到 `persistent-assets/design/_baseline/`（commit 由用户自行决定，不作为进入下阶段的硬门槛）
- [ ] 用户回复确认

### 4.2 Stage 3 内部完整性反查

- [ ] 模块详述章节数 = HLD 模块数（不漏不多）
- [ ] 每个模块都有：概述/依赖/接口/规约/状态机/时序图/数据模型/并发/可观测性/风险 全 10 个子章节
- [ ] 每个对外接口都有五要素（签名/前置/后置/错误码/幂等）
- [ ] 状态机标注 entry/exit/非法事件处理
- [ ] 时序图覆盖全部错误码触发路径
- [ ] 全局错误码表与各接口错误码列表互相一致

### 4.3 Stage 2 → 3 派生一致性

- [ ] **模块清单 = 模块详述**：HLD 模块清单 ⊆ DLD §4.x 详述（不漏）；DLD §4.x ⊆ HLD 模块清单（不多）
- [ ] **依赖关系一致**：HLD §6 模块间通信中每条边在 DLD 各模块 §4.x.2 依赖清单中双向出现
- [ ] **外部依赖一致**：HLD §7 外部依赖表全部在某模块 §4.x.2 中被消费
- [ ] **跨切面落地**：HLD §8 跨切面策略中的具体技术（日志框架、监控指标、鉴权方式）在 DLD 命名/埋点/接口契约中体现
- [ ] **存储一致**：HLD §9.1 聚合根→存储映射与 DLD §4.x.7 数据模型表名/字段一致
- [ ] **NFR 落地**：HLD §11 每条 NFR 在某模块 §4.x.9 可观测性中可被度量

### 4.4 Stage 1 → 3 派生一致性（递归补检）

- [ ] **业务规则 → 行为规约**：Stage 1 §8 每条 R# 在 DLD 某模块 §4.x.4 内部行为规约或 §4.x.5 状态机中体现
- [ ] **聚合根 → 数据模型**：Stage 1 §6 每个聚合根字段在 DLD §4.x.7 中能找到
- [ ] **领域事件 → 时序图**：Stage 1 §7 每个领域事件在 DLD 某模块 §4.x.6 时序图中触发

## 5. 四阶段全部完成后（最终一致性总检）

在 Step 5 交棒前执行：

- [ ] PRD in-scope 全集 ⊆ Stage 4 验收范围
- [ ] DLD 接口全集 ⊆ Stage 4 功能验收用例（happy + 异常皆覆盖）
- [ ] DLD 错误码全集 ⊆ Stage 4 异常用例（错误码维度）
- [ ] DLD 状态机非法转移全集 ⊆ Stage 4 状态机拒绝用例
- [ ] HLD NFR 全集 ⊆ Stage 4 性能/安全/可观测/兼容性用例
- [ ] Stage 1 业务规则全集 ⊆ Stage 4 业务规则用例

任一不通过 → 回 Stage 4 补用例（不要降低判据、不要假装覆盖）。

### 5.M 维护模式末次反查（如本次为维护跑）

在 Step M5 交棒前执行（替代 §5 全量反查，仅对变化部分校验）：

- [ ] 4 份产物头部「变更记录」段均已追加本次条目，与变化清单 1:1
- [ ] 每条变更记录的"影响章节"列在该文件中确实有对应改动（非空写）
- [ ] M1 影响矩阵每个 Stage 列对应的章节实际修改 = 矩阵列出（不漏不多）
- [ ] 修改类变化的具体值（配额 / KPI 阈值 / 错误码描述等）4 份产物保持一致（无版本残留）
- [ ] 废弃类变化对应的派生表追溯条目已删除或标注「v<旧版本>之前生效」
- [ ] 新增类变化的派生链完整：Stage 1 新元素 → Stage 2/3 落地 → Stage 4 验收用例

任一项 ✗ → 回 Step M2 补全。

## 6. 检查执行方式

- 自动化最小化：用 grep/diff 校验文件存在与章节锚点
- 语义检查：通读派生与追溯表，逐行核对
- 模板内置追溯表（每个 stage 模板末尾的"派生与追溯"章节）就是为这套检查准备的——填齐它就完成大半检查
