## 目标

依据已有准出标准，对代码变更和验证证据执行准出审核，输出明确结论：

- `准出`：硬性标准全部通过，无阻塞风险。
- `有条件准出`：硬性标准通过，仅剩已接受的非阻塞风险。
- `不准出`：存在硬性标准失败或证据缺失。

本 Skill 只负责“按规则判定”，不负责重新生成准出标准。缺少准出标准时，先调用 `tac-release-gate-generate`。

## 输入

| 输入 | 必填 | 说明 |
|---|---|---|
| 准出标准文件 | 是 | 默认 `persistent-assets/design/_baseline/04-准出标准.md` |
| 代码目标 | 是 | branch、commit、tag、模块路径或变更范围 |
| 变更证据 | 是 | diff、文件列表、需求完成说明或发布范围 |
| 验证证据 | 建议 | 构建、测试、lint、CI、手动验收、监控或审批记录 |

## Step 0: 标准有效性检查

1. 找到准出标准文件。
2. 运行 `scripts/validate_release_gate.py <标准文件>`。
3. 若文件不存在或结构不完整，不能做权威准出判断；输出 BLOCKED，并建议先使用 `tac-release-gate-generate`。

## Step 1: 判断适用层级

按变更范围选择检查层级：

| 变更范围 | 必查层级 |
|---|---|
| 小功能 / 缺陷修复 | 需求/功能级 + 被触碰的项目/系统级底线 |
| 版本发布 | 版本发布级 + 涉及需求/功能级 + 项目/系统级底线 |
| 架构 / 核心业务变更 | 项目/系统级 + 版本发布级风险 + 需求/功能级完成度 |

不要因为本次是“小改动”就跳过被触碰的系统级硬性约束。

## Step 2: 逐条判定

对每条适用标准给出唯一状态：

| 状态 | 判定条件 |
|---|---|
| `PASS` | 有证据直接满足标准 |
| `FAIL` | 有证据直接违反标准 |
| `BLOCKED` | 标准适用，但缺少必要证据 |
| `N/A` | 标准确实不适用，并有原因 |

PASS 必须引用证据。证据可以是命令输出、测试报告、CI 结果、代码路径、文档段落或人工审批记录。

不能因为“代码看起来没问题”“用户说测过了”“理论上应该满足”就标记 PASS。

审核证据参考 `references/quality-evidence.md`。如果准出标准涉及功能完成度、性能、代码质量、安全、架构、可观测性或兼容性，但缺少对应证据，优先标记 `BLOCKED`，不要降级成建议项。

## Step 3: 输出准出审核报告

使用以下格式：

```markdown
## 结论

准出 / 有条件准出 / 不准出

## 阻塞项

- [FAIL/BLOCKED] <层级> <标准 ID>：<问题>；证据：<证据>；需要动作：<动作>

## 分层检查结果

| 层级 | PASS | FAIL | BLOCKED | N/A |
|---|---:|---:|---:|---:|
| 项目/系统级 | 0 | 0 | 0 | 0 |
| 版本发布级 | 0 | 0 | 0 | 0 |
| 需求/功能级 | 0 | 0 | 0 | 0 |

## 主要风险

- <风险>：影响 <影响>；缓解 <缓解方案>

## 已验证证据

- <命令/文档/代码路径/测试报告>：<结果>
```

## 结论规则

| 结论 | 条件 |
|---|---|
| 准出 | 所有适用硬性标准为 PASS 或 N/A |
| 有条件准出 | 所有适用硬性标准为 PASS 或 N/A，且建议项风险已记录并被接受 |
| 不准出 | 任一适用硬性标准为 FAIL 或 BLOCKED |

## 硬规则

1. **缺标准不准审**：没有准出标准时，不能输出权威准出结论。
2. **缺证据不能通过**：检查模式下缺关键证据必须标为 BLOCKED。
3. **阻塞项优先**：报告先写 FAIL/BLOCKED，再写统计和风险。
4. **只读业务代码**：本 Skill 可读取代码和报告，不修改业务实现。
5. **不执行 git**：不 add、不 commit、不打 tag。
